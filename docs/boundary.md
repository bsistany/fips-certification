## 2. Boundary Definition

FIPS 140-3 requires a precisely defined boundary separating the module from
its operational environment. For a software module the boundary is logical,
not physical.

### Inside the boundary

The following source files and their compiled object files constitute the
cryptographic module:

| Source file | Object file | Role |
|---|---|---|
| `src/aes.c` | `src/aes.o` | AES-128/256 CBC implementation |
| `src/sha256.c` | `src/sha256.o` | SHA-256 implementation |
| `src/hmac.c` | `src/hmac.o` | HMAC-SHA-256 implementation |
| `src/pbkdf2.c` | `src/pbkdf2.o` | PBKDF2-HMAC-SHA-256 implementation |
| `src/fips.c` | `src/fips.o` | FIPS mode flag and algorithm approval |
| `src/self_test.c` | `src/self_test.o` | Power-on self-tests (KATs) |

The corresponding public headers (`include/*.h`) define the interface surface
of the module but contain no executable code. They are considered part of the
boundary definition.

### Outside the boundary

The following are explicitly outside the module boundary:

| Item | Reason |
|---|---|
| Calling application | Operational environment — untrusted by definition |
| `tests/test_*.c` | Test evidence — not shipped as part of the module |
| C standard library (`memcpy`, `memset`, `memcmp`) | Environmental dependency — see Section 5 |
| Compiler stack protection (`__stack_chk_fail`) | Environmental dependency — see Section 5 |
| Operating system, hardware, CPU | Operational environment |

### Rationale for including fips.c and self_test.c

These files do not implement cryptographic primitives but they control access
to them. Compromising either file would allow an attacker to bypass the
security policy of the module. Per FIPS 140-3 the boundary must encompass all
components whose compromise would undermine the module's security guarantees.
Both files therefore belong inside the boundary.

## 3. Interfaces (FIPS 140-3 §7.6)

FIPS 140-3 defines four interface types for a software module. Every
interaction between the module and the outside world must go through one of
these interfaces.

| Interface type | Description | Examples |
|---|---|---|
| Data Input | Plaintext, keys, IVs, salts, passwords passed into the module | `aes_cbc_encrypt(..., in, ...)`, `pbkdf2_hmac_sha256(password, ...)` |
| Data Output | Ciphertext, digests, MACs, derived keys returned by the module | `aes_cbc_encrypt(..., out, ...)`, `sha256_final(..., digest)` |
| Control Input | API calls that direct the module's operation or mode | `fips_mode_enable()`, `fips_self_test_run()`, `aes_init()` |
| Status Output | Return codes and status strings indicating module state | `fips_mode_status()`, `fips_check_algorithm()`, return values |

All data entering and leaving the module passes through the public API
functions listed in Appendix A Section 1. There are no other input or output
paths. In particular the module does not write to files, sockets, or standard
output streams.

## 4. Roles and Services

FIPS 140-3 requires at least one operator role. At Security Level 1
authentication is not required.

| Role | Description |
|---|---|
| Crypto Officer | Enables FIPS mode, triggers self-tests, queries module status |
| User | Calls cryptographic services through the public API |

### Approved services available to the User role

| Service | Functions |
|---|---|
| Symmetric encryption | `aes_init`, `aes_cbc_encrypt`, `aes_cbc_decrypt`, `aes_zeroize` |
| Hashing | `sha256_init`, `sha256_update`, `sha256_final`, `sha256` |
| Message authentication | `hmac_sha256_init`, `hmac_sha256_update`, `hmac_sha256_final`, `hmac_sha256` |
| Key derivation | `pbkdf2_hmac_sha256` |

### Services available to the Crypto Officer role

| Service | Functions |
|---|---|
| FIPS mode control | `fips_mode_enable`, `fips_mode_disable`, `fips_mode_active` |
| Status query | `fips_mode_status`, `fips_check_algorithm`, `fips_self_test_passed` |
| Self-test execution | `fips_self_test_run` |

## 5. External Dependencies (Operational Environment)

These symbols are resolved outside the module boundary. The module's correct
operation depends on them behaving correctly. They are documented here as
environmental assumptions per FIPS 140-3.

| Symbol | Source | Justification |
|---|---|---|
| `memcpy` | C standard library | Required for block and buffer operations |
| `memset` | C standard library | Required for buffer initialisation and zeroization |
| `memcmp` | C standard library | Required for digest comparison |
| `__stack_chk_fail` / `__stack_chk_guard` | Compiler runtime | Stack smashing protection injected by compiler — acceptable at Security Level 1 |

---

## 6. Critical Security Parameters (CSPs)

All CSPs reside in RAM only. No CSPs are written to persistent storage by
the module.

| CSP | Location | Zeroization |
|---|---|---|
| AES key (`AES_CTX.round_keys`) | Stack / caller-allocated struct | `aes_zeroize()` using volatile write |
| HMAC key material (`HMAC_SHA256_CTX`) | Stack | Zeroed in `hmac_sha256_final()` using volatile write |
| PBKDF2 intermediate state (`U`, `T` buffers) | Stack | Zeroed at end of `pbkdf2_hmac_sha256()` using volatile write |
| FIPS mode flag (`fips_mode_on`) | Static storage (`fips.c`) | Module unload |
| Self-test state (`self_test_ok`) | Static storage (`fips.c`) | Module unload |

---

## 7. Known Gaps

The following items are documented as known gaps relative to a full FIPS 140-3
Security Level 1 submission. They would need to be resolved before submitting
to a CMVP lab.

| Gap | Description | FIPS Reference |
|---|---|---|
| `fips_set_self_test_passed` is public | This function should be internal only. As a public symbol it allows a caller to bypass the self-test requirement. | FIPS 140-3 §10.3.1 |
| No software integrity test | POST should include an HMAC-SHA-256 over the module's own compiled binary compared to a reference value embedded at build time. | FIPS 140-3 §10.3.1 |
| No DRBG | An approved Deterministic Random Bit Generator (CTR_DRBG or HMAC_DRBG per SP 800-90A) is required for key generation. | SP 800-90A Rev 1 |
| No CRNGT | Required on every DRBG output — not applicable until a DRBG is implemented. | FIPS 140-3 §10.3.2 |
| No asymmetric algorithms | RSA and ECDSA are not implemented. A PCT would be required on every key generation. | FIPS 140-3 §10.3.3 |
| Memory zeroization not guaranteed | C volatile writes are used but the C standard does not guarantee the compiler will not optimise them away. A platform-specific `explicit_bzero` or `memset_s` would be more robust. | FIPS 140-3 §7.9 |
